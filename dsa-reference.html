✅ Strengths
  • O(1) random access via index
  • Cache-friendly — excellent spatial locality
  • Simple to implement and use
  • Supports binary search on sorted arrays
⚠️ Weaknesses / Trade-offs
  • Fixed size (static) — resizing is expensive O(n)
  • Insertions/deletions in middle are O(n) due to shifting
  • Wasted memory if over-allocated
  • Contiguous memory may be hard to allocate for large sizes
🐛 Corner Cases
  • Off-by-one errors (0-indexed vs 1-indexed)
  • Empty array — accessing index 0 causes crash
  • Integer overflow in mid = (left+right)/2 → use left+(right-left)/2
  • Negative indices in some languages
  • Stack overflow with recursive array traversal on large arrays
🚀 Real-world Applications
  • Database row storage and buffers
  • Image pixel matrices (2D arrays)
  • Stack/Queue implementations underneath
  • Lookup tables, memoization caches
  • NumPy, GPU compute tensors (ML)

✅ Strengths
  • O(1) insertions/deletions at head (singly) or both ends (doubly)

  • No wasted capacity unlike arrays
  • Doubly linked supports reverse traversal
⚠️ Weaknesses / Trade-offs
  • No random access — O(n) to reach index k

  • Null pointer dereference on empty list
  • Single node: head == tail, deletion must update both
  • Cycle detection — Floyd's tortoise and hare
  • Losing head reference during traversal
  • Inserting after last node vs inserting as new tail
🚀 Real-world Applications
  • Browser back/forward navigation (doubly)
  • Music/video playlist management
  • OS process scheduling queues
  • Undo/redo stacks in editors
  • Memory allocators (free list)

Heap (Min / Max)
Complete binary tree, heap property
Insert O(log n) Extract-Min O(log n) Peek O(1)

✅ Strengths
  • O(1) peek at min/max — ideal for priority queues
  • Can be stored compactly as an array (no pointers)
  • Build heap from array in O(n) — better than n insertions
  • Basis for HeapSort (O(n log n) in-place)
⚠️ Weaknesses
  • No O(log n) search for arbitrary element
  • Not cache-friendly like arrays for certain ops
  • Merging two heaps is O(n) unless Fibonacci heap
🐛 Corner Cases
  • Extract from empty heap — underflow
  • Heapify direction: sift-up vs sift-down
  • Equal priority elements — no guarantee on order
  • 0-indexed vs 1-indexed array formula change
🚀 Applications
  • Dijkstra's & A* shortest path algorithms
  • Median maintenance (min + max heap combo)
  • Top-K elements problems
  • OS task schedulers
  • Merge K sorted lists

🚀 Applications
  • Autocomplete (search engines, IDEs)
  • Spell checkers
  • IP routing tables
  • Word games (Boggle, Scrabble solvers)
  • Contact name search on mobile

  • Counting frequencies, anagram detection
  • Two-sum problems, subarray sums
  • Memoization in dynamic programming
  • Language runtime symbol tables

01 Merge Sort
Stable O(n log n) O(n) space
CaseTimeSpace
BestO(n log n)O(n)
AverageO(n log n)
WorstO(n log n)

function mergeSort(arr): if len(arr) <= 1: return arr mid = len(arr) / 2 left = mergeSort(arr[0..mid]) right = mergeSort(arr[mid..]) return merge(left, right) function merge(L, R): result = [] i = j = 0 while i < len(L) and j < len(R): if L[i] <= R[j]: result.append(L[i++]) else: result.append(R[j++]) return result + remaining
✅ Strengths
  • Guaranteed O(n log n) — no bad inputs
  • Stable sort — equal elements keep original order
  • Works well for linked lists (no random access needed)
  • Can be parallelized (divide-and-conquer)
⚠️ Trade-offs
  • O(n) extra space — not in-place
  • Slower than quicksort in practice due to memory allocation
  • Not cache-friendly for large arrays
🐛 Corner Cases
  • Empty or single-element array — base case
  • All equal elements — still O(n log n), stable
  • Already sorted array — no benefit over quicksort
🚀 Applications
  • Sorting linked lists
  • External sorting (data too large for RAM)
  • Stable sort in Java (Arrays.sort for objects), Python
02 Quick Sort
Unstable O(n log n) avg In-place
CaseTimeSpace
BestO(n log n)O(log n)*
AverageO(n log n)
WorstO(n²)

function quickSort(arr, low, high): if low < high: p = partition(arr, low, high) quickSort(arr, low, p-1) quickSort(arr, p+1, high) function partition(arr, low, high): pivot = arr[high] // last element as pivot i = low - 1 for j = low to high-1: if arr[j] <= pivot: i++; swap(arr[i], arr[j]) swap(arr[i+1], arr[high]) return i + 1
✅ Strengths
  • In-place — O(log n) space (recursion stack)
  • Excellent cache performance — works on contiguous memory
  • Fastest in practice for most random inputs
⚠️ Trade-offs
  • Worst case O(n²) on sorted/reverse-sorted — use random pivot
  • Not stable — relative order of equal elements not preserved
  • Difficult to implement correctly without bugs
🐛 Corner Cases
  • All same elements — always triggers worst pivot split
  • Already sorted input with last-element pivot → O(n²)
  • Median-of-three pivot selection mitigates worst case

Searching Algorithms
Linear · Binary · BFS · DFS · Interpolation
01 Binary Search
O(log n) Sorted array required
CaseTimeSpace
BestO(1)O(1)
AverageO(log n)
WorstO(log n)

function binarySearch(arr, target): low = 0; high = len(arr) - 1 while low <= high: mid = low + (high - low) / 2 if arr[mid] == target: return mid elif arr[mid] < target: low = mid + 1 else: high = mid - 1 return -1 // not found
✅ Strengths
  • O(log n) — halves search space each iteration
  • Very simple iterative implementation
  • O(1) space in iterative form
🐛 Corner Cases
  • Integer overflow: use mid = low + (high-low)/2
  • Empty array — low > high at start
  • All same elements with target not equal
  • Duplicate elements — find leftmost/rightmost variant
02 BFS — Breadth-First Search
O(V+E) Shortest path (unweighted)
MetricValue
TimeO(V+E)
SpaceO(V)

function bfs(graph, start): visited = {start} queue = [start] while queue: node = queue.dequeue() process(node) for neighbor in graph[node]: if neighbor not in visited: visited.add(neighbor) queue.enqueue(neighbor)
✅ When to use BFS
  • Shortest path in unweighted graph
  • Level-order tree traversal
  • Finding all nodes within k distance
  • Web crawlers, social network connections
⚠️ Trade-offs vs DFS
  • Uses more memory (entire level in queue)
  • Better for shortest paths; DFS better for cycle detection
  • Not suitable for very deep/wide graphs (memory)
🐛 Corner Cases
  • Disconnected graph — BFS from one node won't reach all
  • Self-loops — mark visited before adding to queue
03 DFS — Depth-First Search
O(V+E) Backtracking / Cycles
MetricValue
TimeO(V+E)
Space (recursive)O(V)

function dfs(graph, node, visited): visited.add(node) process(node) for neighbor in graph[node]: if neighbor not in visited: dfs(graph, neighbor, visited)
✅ When to use DFS
  • Detecting cycles in a graph
  • Topological sorting (DAGs)
  • Finding strongly connected components
  • Maze solving, puzzles, backtracking
  • Tree traversals (pre/in/post order)
🐛 Corner Cases
  • Stack overflow on very deep graphs (use iterative)
  • Must mark visited BEFORE recursing (not after)
  • Back edges indicate cycles in directed graphs

Graph Algorithms
Dijkstra · Bellman-Ford · Floyd-Warshall · Kruskal · Prim · Topological Sort
01 Dijkstra's Algorithm
O((V+E) log V) Shortest Path No negative edges
ImplementationTimeSpace
Min-Heap + Adj ListO((V+E) log V)O(V)
Array (simple)O(V²)O(V)
✅ Key Points
  • Greedy: always process closest unvisited node
  • Uses min-heap (priority queue) for efficiency
  • Finds shortest path from one source to all vertices
  • Fails with negative edge weights — use Bellman-Ford
🐛 Corner Cases
  • Negative weights → incorrect results (use Bellman-Ford)
  • Disconnected graph → some nodes stay at ∞
  • Multiple shortest paths — algorithm finds one
🚀 Applications
  • GPS navigation (Google Maps, Apple Maps)
  • Network routing (OSPF protocol)
  • Game AI pathfinding (with A*)
02 Bellman-Ford Algorithm
O(V·E) Handles negative edges
MetricValue
TimeO(V · E)
SpaceO(V)
Key Idea
  • Relax all edges V-1 times
  • Handles negative weight edges (unlike Dijkstra)
  • Detects negative weight cycles on V-th iteration
  • Slower than Dijkstra but more general
🐛 Corner Cases
  • Negative cycle — report "no shortest path"
  • Disconnected graph — unreachable nodes stay ∞
03 Topological Sort
O(V+E) DAG only
MethodTimeSpace
Kahn's (BFS)O(V+E)O(V)
DFS-basedO(V+E)O(V)

  • Order vertices so all edges go A → B (A before B)

🚀 Applications
  • Build systems (Make, Gradle)
  • Package dependency resolution
  • Course prerequisite scheduling
  • Spreadsheet formula evaluation order

Dynamic Programming
Memoization · Tabulation · LCS · Knapsack · LIS · Coin Change
DP Dynamic Programming — Core Concepts
Optimal substructure Overlapping subproblems
Two Approaches
  • Top-down (Memoization): Recursion + cache results → avoids recomputation
  • Bottom-up (Tabulation): Build up from base cases → fills a table iteratively
  • Both give same time complexity, tabulation has lower stack overhead
  • Space optimization: often reduce 2D table to 1D or a few variables
Classic DP Problems
  • 0/1 Knapsack: O(n·W) time, O(W) space
  • Longest Common Subsequence: O(m·n) time
  • Longest Increasing Subsequence: O(n log n) with patience sort
  • Coin Change: O(amount · coins) time
  • Edit Distance: O(m·n) time
  • Matrix Chain Multiplication: O(n³)
🐛 Common DP Pitfalls
  • Not identifying base cases correctly → wrong answers or index errors
  • Incorrect recurrence relation — the heart of every DP problem
  • Off-by-one in table dimensions (dp[n+1] vs dp[n])
  • Assuming overlapping subproblems exist when they don't (greedy may be simpler)
  • Not resetting memo/table between test cases in competitive programming
