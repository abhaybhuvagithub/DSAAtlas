Software Architecture Techniques
Quality attributes · Patterns · Integration · Reliability
🧱
Layered (N-tier) Architecture
Presentation → Service → Domain → Data access
Simple governance Can become tangled Common baseline
DimensionGuidance
Best forCRUD-heavy apps, clear separation of concerns, stable domain
Trade-offsLayer bypassing, coupling via schema, “god service layer” risk
PitfallsAnemic domain, leaking persistence details, over-shared utilities
PracticesExplicit boundaries, DTOs, dependency inversion, modular monolith
Boundary rules
  • UI depends on Services; Services depend on Domain
  • Domain depends on no infrastructure (ports/interfaces)
  • Persistence behind Repository/DAO
Testing strategy
  • Domain unit tests (fast)
  • Service tests with mocks/doubles
  • Integration tests at DB/API boundaries
🧩
Microservices
Independent deployable services around bounded contexts
Independent scaling Ops complexity Distributed failures
DimensionGuidance
Best forLarge teams, high change rate, clear domain boundaries, separate scaling
Trade-offsLatency, debugging overhead, consistency challenges
PitfallsShared DB, chatty sync calls, “distributed monolith”
EnablersObservability, CI/CD, contract testing, SRE practices, ownership
  • Do you have logs/metrics/traces + alerting?
  • Independent deployability (CI/CD, feature flags)?
  • Timeouts, retries, idempotency, circuit breakers?
  • Data ownership + eventual consistency strategy?
  • Clear ownership and on-call model per service?

Event-Driven Architecture (EDA)
Publish events; consumers react asynchronously
Decoupling Scalable fanout Eventual consistency
ConceptDetails
PatternsPub/Sub, event streams, outbox, sagas
DeliveryAt-most-once / at-least-once; “effectively once” via idempotency

Trade-offsHarder debugging; replay/reprocessing complexity
Outbox pattern
// In same DB transaction: write business_row append outbox_event(type, payload, id) // Relay worker: poll outbox where status=NEW publish to broker mark outbox status=SENT
Idempotent consumers
  • Use eventId dedupe table / cache
  • Upserts instead of blind inserts
  • Idempotency keys for external side effects

SD-01 URL Shortener
High QPS reads Cache-heavy Abuse control
Requirements
  • Shorten long URL → short code
  • Redirect short code → original URL (very high read traffic)
  • Optional: custom alias, expiry, click analytics
  • Non-functional: low-latency redirect, high availability
APIs
POST /v1/links { longUrl, customCode?, expiry? } → { code, shortUrl } GET /{code} → 301/302 redirect to longUrl GET /v1/links/{code}/stats → { clicks, recent, referrers... }
Data model
links( code PK, long_url, created_at, expires_at, owner_id?, status ) click_events(code, ts, ip_hash?, ua_hash?, ref?) (or aggregated counters per minute/hour)
Key design choices
  • Code generation: base62(snowflake/id) or random + collision check
  • Redirect path: CDN/edge → app → cache → DB
  • Cache: code→longUrl in Redis with TTL (expiry aware)
  • Analytics: async via stream/queue; aggregate nearline/offline
  • Abuse: per-IP rate limit, URL reputation checks, allow/deny lists
AreaApproach
ScalingReads dominate: cache + edge + stateless redirect service
ConsistencyStrong for create; eventual for analytics counters
Failure modesCache miss storms → request coalescing; DB hot partitions → sharding
SecurityInput validation, SSRF protection, phishing controls, audit logs
SD-02 News Feed / Timeline
Fanout Ranking Hot users
Requirements
  • Users post content; followers see a feed
  • Low-latency feed reads; pagination
  • Handle “celebrity” accounts (millions of followers)
  • Ranking signals (recency, affinity, engagement)
Core approaches
  • Fanout-on-write: push post IDs into follower inboxes
  • Fanout-on-read: merge at read-time from followees
  • Hybrid: push normal users, pull celebrities + cache
Data model (simplified)
posts(post_id PK, author_id, ts, content_ref,...) follows(user_id, followee_id) inbox(user_id, ts, post_id) // materialized feed (or Redis sorted set per user)
Ranking
  • Two-stage: candidate generation → ML ranking
  • Cache ranked results; incremental refresh
  • Spam/quality filters pre-ranking
TopicDesign notes
Hot keysCelebrity fanout avoided via pull + caching + batching
StorageInbox sharded by user_id; time-ordered indexing
ConsistencyEventual acceptable; preserve per-user ordering when possible
ReliabilityQueue backpressure, retries, idempotent fanout workers
SD-03 Real-time Chat (1:1 / Groups)
WebSockets Ordering Offline delivery
Requirements
  • Send/receive messages in real time
  • Delivery/read receipts (optional)
  • Offline storage + multi-device sync
  • Group membership and fanout
Protocol
  • Clients connect via WebSocket to a gateway
  • Send message → server assigns msg_id + sequence
  • Ack flow: delivered/read (with idempotency)
Data model
conversations(conv_id, type, created_at) members(conv_id, user_id, role) messages(conv_id, seq, msg_id, sender, ts, payload_ref) user_inbox(user_id, conv_id, last_read_seq, last_delivered_seq)
Key design points
  • Ordering: per-conversation sequence numbers (conv_id partition)
  • Fanout: online sockets; offline inbox store otherwise
  • Presence: separate service with TTL heartbeats
  • Media: object storage; messages store references
ConcernApproach
ScalabilityShard by conv_id; stateless gateways; optional sticky sessions
ReliabilityAt-least-once with idempotent writes + dedupe
SecurityAuthn on connect; authz per conversation; optional E2EE
